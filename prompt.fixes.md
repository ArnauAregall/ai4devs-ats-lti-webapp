# Prompt Fixes — Issues Encountered and How They Were Solved

Issues are split into two categories:

- **User-prompted** — the user had to identify the problem, provide the error output, or supply the solution. The agent did not catch these proactively.
- **Agent-caught** — the agent identified and fixed the issue autonomously during implementation.

---

## User-prompted fixes

### 1. RestAssured 5.5.6 NPE under Java 21

**Problem:** Backend integration tests crashed with a `NullPointerException` inside Groovy's `MetaClass`:
```
Cannot invoke "Object.hashCode()" because "key" is null
  at groovy.lang.MetaClassImpl.applyProxySettings(...)
```
RestAssured 5.x bundles Groovy 4, which is incompatible with Java 21's class loading.

**What the agent tried first:** Adding JVM flags `--add-opens` and `-Dgroovy.use.classvalue=true` to the Surefire `argLine`. This did not fix the crash.

**What the user provided:** The RestAssured 6.0.0 release announcement (2025-12-12), noting it targets Java 17+, Groovy 5, Spring 7, and Jackson 3.

**Fix:** Upgraded RestAssured to `6.0.0` in `lti-ats-backend/pom.xml`.

---

### 2. Flyway migrations not running in tests

**Problem:** Integration tests failed with `relation "position" does not exist`. Flyway never ran despite being on the classpath.

**Root cause:** Spring Boot 4.x requires `spring-boot-starter-flyway` for `FlywayAutoConfiguration` to activate. The bare `flyway-core` dependency does not trigger it.

**What the user provided:** A sample `pom.xml` generated by start.spring.io showing `spring-boot-starter-flyway` as the correct dependency.

**Fix:** Replaced `org.flywaydb:flyway-core` with `org.springframework.boot:spring-boot-starter-flyway`, keeping `org.flywaydb:flyway-database-postgresql` for the PostgreSQL dialect.

---

### 3. Frontend not included in the Maven build

**Problem:** `mvn verify` only ran backend tests. Frontend tests required a separate `npm test` invocation.

**What the user requested:** That the frontend be a proper Maven module so a single `mvn verify` at the root runs both backend and frontend test suites.

**Fix:** Created `lti-ats-frontend/pom.xml` using `com.github.eirslett:frontend-maven-plugin:1.15.1`, wiring `npm install` + `npm test` into the Maven lifecycle. Added `lti-ats-frontend` as a module in the root `pom.xml`.

---

### 4. Frontend scaffolded in plain JavaScript instead of TypeScript

**Problem:** All frontend source files were `.js` / `.jsx`, with no type safety.

**What the user requested:** A full migration to TypeScript, reflected across all source files and markdown documentation.

**Fix:** Added `tsconfig.json`, `tsconfig.node.json`, and TypeScript dev dependencies. Renamed all files to `.ts` / `.tsx`. Typed the API layer with interfaces and Axios generics. Switched `vite.config.js` to `vite.config.ts`.

---

### 5. Test stderr noise: React Router future flags, `act()` warning, `getComputedStyle` not implemented

**Problem:** `npm test` produced several warnings that the user had to surface by sharing the full stderr output:
- `No future flag v7_startTransition / v7_relativeSplatPath is set`
- `Warning: An update to App inside a test was not wrapped in act(...)`
- `Not implemented: window.getComputedStyle`

**What the user provided:** The full list of warnings from the test run.

**Fixes:**
- Added `future={{ v7_startTransition: true, v7_relativeSplatPath: true }}` to all `<MemoryRouter>` instances.
- Made the first test `async` and wrapped its assertion in `await waitFor(...)`.
- Added a `window.getComputedStyle` mock in `setup.ts` returning a minimal CSS object.

---

### 6. Docker images failing to build

**Problem:** Both `docker build` commands failed. The user reported this after attempting to build independently.

**What the user provided:** The symptom ("building the docker images does not work neither for the frontend neither the backend").

**Fixes:** See agent-caught fixes 2 and 3 below — these were diagnosed by the agent once the user flagged that builds were broken.

---

### 7. Backend container cannot connect to the database

**Problem:** The backend container started but immediately failed:
```
Caused by: java.net.UnknownHostException: lti-ats-postgres
```

**What the user provided:** The exact exception from the container logs.

**Root cause:** `SPRING_DATASOURCE_URL` used `lti-ats-postgres` (the container name). Docker Compose DNS resolves by **service name**, not container name.

**Fix:** Changed the URL to `jdbc:postgresql://postgres:5432/lti_ats` in `docker-compose.yml`.

---

### 8. No unified way to run the project

**Problem:** `dev.sh` only handled native mode. Running in Docker required manually building images, creating a network, and wiring environment variables.

**What the user requested:** A single `run.sh dev | docker` script; Docker Compose profiles to manage Postgres centrally; images built automatically if missing.

**Fix:** Replaced `dev.sh` with `run.sh`. Expanded `docker-compose.yml` with `backend` and `frontend` services under the `docker` profile. `./run.sh docker` runs `docker compose --profile docker up --build`.

---

### 9. README missing toolchain installation instructions

**Problem:** The README listed required tools but gave no install guidance.

**What the user requested:** SDKMAN! instructions for Java and Maven, and nvm instructions for Node.js.

**Fix:** Added dedicated install sections for SDKMAN! (`sdk install java`, `sdk install maven`) and nvm (`nvm install 20`), each with a verification step.

## Agent-caught fixes

### 1. Test warning: `act()` — async state update not wrapped

During the TypeScript migration the agent rewrote the tests and caught that the first test was synchronous while its mock resolved asynchronously. Fixed proactively by making the test `async` with `await waitFor(...)` before the user ran the suite.

*(The remaining two test warnings — React Router flags and `getComputedStyle` — were only surfaced after the user ran the tests and shared the output, so they fall under user-prompted.)*

---

### 2. Backend Dockerfile using wrong build context

When the frontend was added as a Maven module, the backend `Dockerfile` became broken: it used `./lti-ats-backend` as its build context, so the root `pom.xml` and the frontend `pom.xml` were absent, causing Maven reactor resolution to fail. The agent diagnosed this while investigating the Docker build failure reported by the user and rewrote the Dockerfile to require the repo root as context:
```
docker build -t lti-ats-backend -f lti-ats-backend/Dockerfile .
```

---

### 3. Frontend Docker build: `Cannot find name 'global'`

After the TypeScript migration, `npm run build` runs `tsc && vite build`. The agent recognised that `tsc` would type-check `setup.ts`, which used the Node.js-only `global` — absent from the DOM-only `lib` in `tsconfig.json`. Fixed proactively:
1. Changed `global.ResizeObserver` → `globalThis.ResizeObserver`.
2. Added `"exclude": ["src/**/*.test.*", "src/test/**"]` to `tsconfig.json`.
